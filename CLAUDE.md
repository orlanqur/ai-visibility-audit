# AI-Visibility Audit System — контекст для Claude

Этот репозиторий — **система для аудита видимости бренда в AI-моделях** (Alice, ChatGPT, Google AIO, Perplexity, Yandex Neuro и т.п.). Один репозиторий = система + много аудитов, каждый аудит = своя папка в `projects/`.

## Структура

```
scripts/                        общие скрипты (режут, считают, строят карту ссылок)
.claude/
  commands/                     slash-команды (/blind-analysis, /cross-model, /link-map)
  skills/
    audit-split-by-model/       параллельный разбор по моделям через subagents
    pipeline-guide/             помогает коллеге с «что делать дальше»
CLAUDE.md                       (этот файл)
README.md                       пошаговая инструкция для пользователя
mcp.json                        подключение docs-intelligence-mcp
projects/
  <brand>/                      отдельный аудит (может быть несколько)
    responses/                  сырые {Model}-{Brand}.md (вход пайплайна)
    sliced/                     нарезка q-файлов с YAML (выход slice.py)
    metrics/                    metrics.csv + link_map.json + link_consensus.csv
    reports/                    сохранённые отчёты (/cross-model складывает сюда)
    .docs-intel/                векторный индекс MCP (создаётся docs-intel index)
```

## Определение активного проекта

Почти все команды и скиллы работают в контексте **одного активного проекта**. Правило:

1. Если в `projects/` одна подпапка — это активный проект.
2. Если несколько — **обязательно спроси у пользователя** имя проекта перед любыми действиями.
3. Если пользователь назвал проект в запросе ("разбери alfa_bank", "по tinkoff") — используй его.

Не предполагай проект молча. Не работай с корнем репо — всё всегда внутри `projects/<active>/`.

## Что в данных

**`projects/<brand>/sliced/<Model>/qNNN.md`** — YAML-frontmatter + `## Вопрос` / `## Ответ` / `## Источники`:
```yaml
model: Alice
brand: Alfa_bank
question_id: 1
question_text: "..."
sources_count: 11
```

**`projects/<brand>/metrics/metrics.csv`** — колонки per вопрос:
- `answer_length_chars`, `answer_length_words` — объём ответа
- `sources_count` — число источников в списке
- `brand_in_body` / `brand_in_sources` — (bool) упомянут ли бренд
- `brand_position_in_sources` — позиция первого бренд-URL в списке (пусто если нет)

**`projects/<brand>/metrics/link_map.json`** — per домен: модели (set), вопросы (set), URL-ы, число упоминаний в теле и источниках.

**`projects/<brand>/metrics/link_consensus.csv`** — плоская таблица доменов с ≥N моделей.

## Терминология

- **blind question** — бренд **не** упомянут в формулировке вопроса. Модель сама выбирает источники. Самый ценный сигнал видимости.
- **branded question** — бренд прямо в формулировке. Модель почти всегда цитирует сайт бренда. Менее показательно.
- **консенсус** — домен, встречающийся у ≥ N моделей. Показывает, какие сайты AI считает авторитетными в нише независимо друг от друга.

## Правила работы

1. **Не цитируй URL или факты из `docs_search` напрямую.** Семантический поиск даёт координаты — прочитай найденный файл через Read, только потом цитируй.

2. **Для кросс-модельных задач** («разбери X по всем моделям», «сравни как модели отвечают на Y») — вызывай skill `audit-split-by-model`. Он разнесёт работу по subagents, каждый со своим чистым контекстом. Не перечитывай `sliced/` сам.

3. **Для типовых задач есть slash-команды**:
   - `/blind-analysis` — разбор слепых вопросов по одной модели
   - `/cross-model` — кросс-модельный синтез (вызывает skill)
   - `/link-map` — карта ссылок + консенсус

4. **Метрики — читай CSV/JSON, не пересчитывай.** На вопрос «у каких моделей бренд не упомянут в q17» — открывай `projects/<brand>/metrics/metrics.csv`, не нарезку.

5. **Навигация — MCP, чтение — Read.** `docs_url_inventory`, `docs_find_by_entity` дают координаты файлов; конкретные цитаты берутся только из реальных sliced-файлов.

## Для первой сессии (онбординг Катерины / нового пользователя)

Если это первая работа с репо — скажи пользователю, что он может в любой момент написать:

> «Что мне делать дальше?» / «какой следующий шаг?» / «помоги с пайплайном»

и ты запустишь skill `pipeline-guide`. Он сам посмотрит состояние проекта (есть ли responses, прогнался ли slice, есть ли metrics, проиндексирован ли MCP) и подскажет следующий шаг.

Если пользователь растерян — не пересказывай README заново, просто вызови `pipeline-guide`.
